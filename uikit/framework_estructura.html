Detalle de la carpeta JS:

Por motivos de optimimización todo el JS se cargará de manera asíncrona excepto aquel que sea necesario para formar la página. Por este motivo hay que ser consciente de que la página se vea bien tanto con JS como sin JS cargado.

Trabajaremos con archivos de “código fuente” que generarán 2 archivos compilados para su uso final en la maqueta (async.js y critical.js).

Estos archivos se generan automáticamente cargando todos los js presentes en sus correspondientes carpetas, para evitar que un JS se añada bastará con cambiar su extensión a .js_, en todo proyecto habrá que hacerlo para evitar cargar JS que no se necesite.

Detalle de alguno de los archivos LESS:

colours.less Todos los colores que se usen en el proyecto han de estar recogido dentro de variables dentro de un grupo llamado paleta, nunca como código de color directamente. De esta manera nos aseguramos que siempre usamos el mismo color y no variaciones que puedan darse por errores en la fase de diseño.

La nomenclatura siempre será en inglés y con este formato:

@color-variación : #355634;

Ejemplo de una gama completa para un mismo color:

@color-grey-softest
@color-grey-soft
@color-grey
@color-grey-dark
@color-grey-darkest

A su vez hay que repetir la declaración de los colores dentro de un array para generar helpers de texto y background:

@array-colors:
    color-white #fff,
    color-success #71a67b,
    color-info #61a0b9,
    color-warning #f08400,
    color-danger #ec3457,
    color-text #4d4d4d,
    color-prueba red

Esto nos genera clases de este estilo:

.text-color-danger { color: @text-color-danger !important  }
.bg-color-danger { background-color: @text-color-danger !important }

En proyectos grandes es probable que tengamos paletas de color adicionales a la global, en ese caso indicamos en la variable a que paleta pertenece añadiendo su nombre delante:

@paleta-color-variación : #355634;
